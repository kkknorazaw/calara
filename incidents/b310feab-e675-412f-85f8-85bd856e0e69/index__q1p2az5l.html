Pepe Claim Reentrancy

1. Incident Overview TL;DR

In BSC transaction 0x5d1ab09dfc3b1250980f8f5f6e573ec315f48728941424f03620304400fa9560 at block 88100768, an attacker borrowed 51 WBNB through a Pancake flash swap, converted it to native BNB, and exploited Pepe at 0x9292a7d96389d94caf41296c2a226ad60ee0e347. The attacker called createClaim(uint256) once directly and once reentrantly through the ERC1155 receiver hook exposed by the claim-token contract 0xeb3c17d804dc1caf163774f012b8f2d883b0fa51.

The root cause is a checks-effects-interactions violation in createClaim. Pepe mints the ERC1155 claim token before it stores the claim tuple and before it increments the claim counter used by nextClaimHash(). The receiver hook therefore reenters while slot 0x0c is still unchanged, causing two ERC1155 units to share one token id. The inner call stores a smaller claim first, the outer call overwrites that same shared record with the larger claim, and both ERC1155 units later redeem against that larger stored amount. The transaction drained 6805552132543554940 wei from Pepe and left the adversary EOA with 6677574452234124600 wei net profit after gas.

2. Key Background

The token id for a new claim is derived from nextClaimHash(). The collected disassembly and state diff show that this derivation depends on Pepe storage slot 0x0c, which was 229 before the transaction and became 231 after two nested claim-creation paths completed. Reusing slot 0x0c before it advances therefore reuses the same claim token id.

The exploit used three adversary-controlled accounts visible on-chain:

  • EOA 0xfd94b7ed68fb9754a13033deb590a23eac26b4f0 submitted the seed transaction and received the final profit.
  • Contract 0xb21384554933a280a041b77d0b9d68967d5bcdec received the flash-swapped WBNB, implemented the ERC1155 receiver hook, reentered Pepe, and repaid the flash swap.
  • Contract 0x24c881b8bc024d7a97709786b2d559079ba01923 received one duplicate claim token and performed the first redemption.

3. Vulnerability Analysis & Root Cause Summary

The vulnerability is an application-layer reentrancy bug in Pepe claim creation, not an MEV-only opportunity or an access-control issue. Pepe computes the new token id, calls the external ERC1155 mint, and only afterward writes the claim state and increments the counter that should make the next id unique. That call order exposes a partially initialized claim to an attacker-controlled receiver hook. Because the reentrant call observes the same pre-increment counter, it mints a second ERC1155 unit with the same token id. The inner call writes one claim tuple, then the outer call resumes and overwrites the same storage slots with a larger quantity and a different unlock time. collectClaim(uint256) later trusts the token-id keyed record without defending against duplicated supply for the same claim id. The broken invariant is that one redeemable claim id should correspond to one funded claim tuple and one redeemable supply position unless the backing record tracks proportional payout.

4. Detailed Root Cause Analysis

The seed trace and receipt establish the duplicated token-id mint directly. In the transaction receipt, two TransferSingle events from the claim-token contract mint the same id 0x707b82055b205da02df65caa800169d981d9fda908005c879eb937a92b4eafa6 to the attacker receiver contract:

TransferSingle(... to=0xb21384554933a280a041b77d0b9d68967d5bcdec,
  id=0x707b82055b205da02df65caa800169d981d9fda908005c879eb937a92b4eafa6, value=1)
TransferSingle(... to=0xb21384554933a280a041b77d0b9d68967d5bcdec,
  id=0x707b82055b205da02df65caa800169d981d9fda908005c879eb937a92b4eafa6, value=1)

The validator rerun log shows the full nested sequence:

Pepe::createClaim{value: 28718355055349438421}(5933277)
  PepeMultiToken::mint(attacker, sharedTokenId, 1)
    attacker::onERC1155Received(...)
      Pepe::createClaim{value: 21912802922805883481}(7776000)
        PepeMultiToken::mint(attacker, sharedTokenId, 1)

The collected Pepe disassembly shows why this is possible. Around 0x2a05, createClaim performs the external call that mints the ERC1155 token. Only after that call returns do the shared claim-state writes occur at 0x2a48, 0x2a52, and 0x2a5c. The claim counter update follows later at 0x2bee:

002a05 ... CALL
002a48 SSTORE
002a52 SSTORE
002a5c SSTORE
...
002bee SSTORE

That ordering matches the state evidence. The pre-state diff shows slot 0x0c = 229. The post-state diff shows slot 0x0c = 231, proving that two increments happened only after both nested createClaim paths had used the same pre-increment value. The same diff also shows the final shared claim record for the duplicated id:

{
  "0x0e959fbbe65c1e3ab9a6502ae1ba0d97520a344ff0eba0bff849271394d84c9a": "0x0000000000000000000000000000000000000000000000018e8c17c437a597d4",
  "0x550f3f840306944cdda81ddc80713db6030246b5b770af22110276ef0705b4ad": "0x000000000000000000000000000000000000000000000000000000006a1a9629"
}

Those values correspond to the larger outer claim quantity 28718355055349438420 and the outer unlock timestamp. Receipt logs confirm two NFTGemClaimCreated events for the same token id with different durations and quantities, first the smaller inner claim and then the larger outer claim. Because the outer call wrote last, the shared mapping entry ended in the larger state.

The exploit completes when both ERC1155 units are redeemed separately. The attacker transfers one unit to helper contract 0x24c881b8bc024d7a97709786b2d559079ba01923 and calls collectClaim(tokenId) from that helper. The attacker receiver then calls collectClaim(tokenId) again on the remaining unit. The receipt contains two redemption events for the same token id, and each redemption amount equals the larger overwritten quantity:

NFTGemClaimRedeemed(... redeemer=0x24c881b8bc024d7a97709786b2d559079ba01923,
  tokenId=0x707b...4eafa6, amount=28718355055349438420)
NFTGemClaimRedeemed(... redeemer=0xb21384554933a280a041b77d0b9d68967d5bcdec,
  tokenId=0x707b...4eafa6, amount=28718355055349438420)

5. Adversary Flow Analysis

The adversary executed the entire ACT flow inside one public BSC transaction.

  1. The EOA 0xfd94b7ed68fb9754a13033deb590a23eac26b4f0 deployed the attacker receiver and initiated the transaction.
  2. The receiver borrowed 51 WBNB from Pancake pair 0x16b9a82891338f9ba80e2d6970fdda79d1eb0dae and unwrapped it to native BNB.
  3. The receiver computed the next claim id from Pepe and called createClaim(5933277) with 28718355055349438421 wei.
  4. During the claim-token mint, the receiver hook onERC1155Received reentered Pepe and called createClaim(7776000) with 21912802922805883481 wei before the outer path advanced the claim counter.
  5. The nested call minted a second unit of the same claim-token id and stored the smaller claim. The outer call then resumed and overwrote the shared claim record with the larger amount and outer unlock time.
  6. The attacker deployed a helper contract, transferred one duplicate ERC1155 unit to it, and redeemed that unit once through the helper and once through the attacker receiver.
  7. Both redemptions paid the larger stored amount. The attacker then wrapped enough BNB back into WBNB to repay 51127819548872180452 wei to the Pancake pair and forwarded the residual BNB to the originating EOA.

The validator rerun log reproduces these steps on a fresh fork with attacker-local contracts rather than the incident addresses, which confirms that the opportunity was permissionless and reproducible.

6. Impact & Losses

Pepe lost 6805552132543554940 wei of native BNB-equivalent value in the incident transaction. The balance-diff artifact shows Pepe falling from 6805555555555555512 wei to 3423012000572 wei, while the attacker EOA increased by 6677574452234124600 wei after gas.
